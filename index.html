My Featured Projects

Mortgage Seller and Broker Risk Tracking Application

Role: Business Analyst, UAT Coordinator, Project Lead

  • Challenges:
    • Manual broker tracking spreadsheets created inefficiencies and compliance risks.
    • Data was fragmented across multiple platforms, causing downstream applications to be inefficient and reducing team productivity.
    • User data entry errors introduced inaccuracies and further slowed processes.
    • Difficult reporting processes limited visibility and timely decision-making.
  • Action: Directed end‑to‑end delivery of a new risk tracking application.
    • Documented current/future state requirements and translated them into actionable functional specs.
    • Coordinated UAT and E2E testing, developing a reusable test case template with built‑in formulae, filters, and TOC for quick reference. Authored and proctored dozens of test cases.
    • Validated data migration, mapping schema/table/column information between old and new databases. Where direct mappings didn’t exist, provided SQL‑based derivations and documentation.
    • Created a SQL‑driven Production Database Meta Data Dictionary in Power BI with slicers (filters) by schema, table, column, and foreign key.
    • Built a Work Queue Report in Power BI, enabling slicing by due date, associate, task, seller, and broker.
    • Established a comprehensive Data Dictionary to accelerate SQL query development and Power BI reporting.
    • Streamlined backlog management by introducing tagging conventions in Azure DevOps.
  • Outcome:
    • Eliminated reliance on spreadsheets, reducing manual effort by 80%
    • Established a single source of truth for seller/broker data, financials, and licensing
    • Streamlined bug tracking and user story management in Azure DevOps
    • Developed UAT best practices, reducing turnaround time by 50%
    • Improved reporting accuracy and speed through SQL‑based Power BI artifacts
    • Launched with zero downtime and seamless adoption across business units
  • Tools: Microsoft Office, SharePoint, Azure DevOps, SQL Server Management Studio, Power BI, Microsoft Copilot

    • Example: Field Definitions Table contrasting ambiguous and useful definitions
      • Object Ambiguous Definition Useful, Descriptive Definition
      addressList Address data An object that contains details about the client's address.
      addressList.clientName Client name Name of the client.
      addressList.address1 Address First line of the client's address.
      Maximum: 55 characters.
      addressList.address2 Address Additional address line, if needed. Examples: apartment number, PO box, care of address.
      Maximum: 55 characters
      addressList.country Country name Name of the country of the client's address. Country codes are defined by the International Organization for Standardization (ISO).
      Maximum: 3 characters.
      addressList.postalcode zipcode Five-digit postal code for a U.S. location. Note: U.S. postal codes may also contain a hyphen and the four-digit extension.
      Maximum: 20 characters.
      addressList.state State Name of the city of the client’s address.
      Maximum: 50 characters.
      languageList Language data An object that contains details about the client's preferred language and any other languages used by the client’s employees.
      languageList.languageCode Language code A numeric code that indicates the client's language preference.
      Maximum: 2 characters.
      languageList.languageUse Indicates language used A numeric code that indicates the general acceptance of each language by the client’s employees.
      • 1: Considered the native language of many or most employees.
      • 2: Used less often but still spoken by many employees.
      • 3: Used infrequently.
      clientAffiliation Client affiliation data An object that contains information about the relationship between the client and ABC Corp.
      clientAffiliation.billingPrefID ID of preferred billing The system-generated identifier that represents a billing preference record.
      clientAffiliation.effectiveDate Effective date Start date of the affiliation record for the client and XYZ Corp.
      Format: YYYY-MM-DD
      clientAffiliation.expirationDate Expiration date End date of the affiliation record for the client and XYZ Corp.
      Format: YYYY-MM-DD
      data Data An object that contains information about the contract.
      data.holdStatus Defines holds on contract Indicates whether the client has any holds currently applied to the contract.
      Valid values:
      • 0: False (default)
      • 1: True
